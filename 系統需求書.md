# 系統需求書 (System Requirements Document) - 全自動化有聲書量產系統

基於「AI 沉浸式有聲圖書館」企劃，本文件定義了全自動化有聲書產出流水線的系統需求，明確指出資料來源、儲存方式、處理工具及最終產出，以確保整體流程能無人值守地自動化運行。

---

## 一、 系統總體目標
建立一條高穩定、高擴展性的全自動流水線：從網路上抓取小說文本，經過自動清理與 AI 語音合成，最終與動態視覺素材結合，產出符合 YouTube 營利標準的長篇有聲書影片，全程無需人工介入。

---

## 二、 資料獲取模組 (Data Sourcing)
負責從外部來源穩定地將生肉文本抓取回來。

*   **獲取來源**：初期針對特定小說網站（以《凡人修仙傳》為 POC）。系統設計需具備 Adapter 擴充性，以便未來無縫接入其他小說網。
*   **獲取工具/技術**：
    *   **核心腳本**：Python。
    *   **靜態網頁**：`requests` + `BeautifulSoup` 或 `lxml`。
    *   **動態網頁 (若需)**：`Playwright` 或 `Selenium`。
*   **自動化保證 (防斷與防封)**：
    *   **狀態與進度紀錄 (斷點續傳)**：將爬取進度（最新成功爬取的章節）即時寫入本地 `progress.json` 狀態檔。中斷重啟時，腳本自動讀取進度，跳過已抓取的章節繼續抓取。
    *   **防封鎖機制**：實作隨機延遲 (Random Delay 2~5秒)、自動輪換 User-Agent，並加入遇到錯誤時的指數退避 (Exponential Backoff) 最多 3 次的重試機制。

---

## 三、 資料處理與中介儲存 (Data Processing & Storage)
負責將抓取回來的雜亂資料，整理並轉換為音檔與影像素材。

*   **專案工作區 (Workspace)**：`G:\我的雲端硬碟\作品\有聲小說\Workspace\<書名>\`

### 1. 文本處理 (Text Processing)
*   **檔案儲存結構**：
    *   **原始文本 (Raw Text)**：從網頁抓取下來未經處理的文字，存放在 `Workspace\<書名>\RawText\`。第一行為章節標題（供後續影片標題卡使用）。
    *   **清理後文本 (Clean Text)**：經過處理準備送給 TTS 的文字，存放在 `Workspace\<書名>\CleanText\`。
*   **檔案命名規則**：`<書名>_chapter_<N>_raw.txt` / `<書名>_chapter_<N>_clean.txt`
*   **儲存格式**：標準 `.txt`，UTF-8 編碼。
*   **處理工具**：Python (`re` 正則表達式模組)。
*   **自動化處理與清理動作**：
    *   **明確排除干擾文字**：利用正則表達式強制過濾網站版權宣告、導覽列文字（如「請記住本站域名:」、「快捷鍵: 上一章」、網站名稱廣告詞等）。
    *   自動過濾 HTML 標籤、亂碼與無意義符號。
    *   **智慧長句截斷**：將過長段落優先依據全形標點 (。！？) 進行截斷，必要時輔以 (，、) 截斷，嚴格限制每句不超過 100 字，並以「每行一句」格式儲存。

---

### 2. 語音生成 (TTS Engine)

#### 主要引擎：Microsoft Edge TTS（`tts_ms.py`）
*   **技術方案**：使用 `edge-tts` Python 套件呼叫 Microsoft Azure 神經語音（預設聲線：`zh-CN-YunxiNeural`）。
*   **Spyder 相容性**：因 Spyder/IPython kernel 本身已有執行中的 asyncio event loop，需在模組頂部套用 `nest_asyncio.apply()` 以允許巢狀呼叫。
*   **處理流程**：
    1.  讀取整章 CleanText（一個 `.txt` 整章送出），edge_tts 一次產出整章音訊。
    2.  edge_tts 直接輸出 `.mp3`（暫存），再用 FFmpeg 轉成 `.wav`。
    3.  刪除暫存 `.mp3`，保留最終 `.wav`。
*   **輸出命名**：`<書名>_chapter_<N>.wav`（**無 part 編號**）

#### 備用引擎：GPT-SoVITS（`tts.py`）
*   **技術方案**：本地部署 GPT-SoVITS，透過其 REST API（`http://127.0.0.1:9880`）逐段呼叫。
*   **處理流程**：
    1.  逐行讀取 CleanText，每行呼叫一次 API，產出暫存 part 檔：`<書名>_chapter_<N>_tmp_part_<NNN>.wav`。
    2.  所有 part 完成後，用 FFmpeg `concat demuxer` + `-c copy`（無損、瞬間）合併成單一章節 WAV。
    3.  刪除全部 `_tmp_part_` 暫存檔，保留最終合併檔。
*   **輸出命名**：`<書名>_chapter_<N>.wav`（**無 part 編號**，與主引擎一致）
*   **容錯**：單次 API 失敗最多重試 3 次，不影響其他章節。

#### 音訊儲存路徑
```
Workspace/<書名>/Audio/<書名>_chapter_<N>.wav
```

---

### 3. 視覺素材獲取
*   **獲取來源/工具**：
    *   **靜態 (本地圖庫輪播)**：支援讀取 `Images/` 資料夾內的多張圖片（非 title_card 圖）。
    *   **動態 (進階)**：未來可擴充呼叫 Stable Diffusion API 或 Midjourney API。
    *   **章節標題卡（自動產生）**：`image_gen.py` 在 TTS 完成後，用 **Pillow** 自動為每章產生一張 1280×720 的標題卡圖片（深色漸層背景 + 書名金字 + 章節標題白字）。
*   **中介儲存**：
    *   章節標題卡：`Workspace/<書名>/Images/<書名>_chapter_<N>.jpg`（由 image_gen.py 自動產生，約 0.1 秒/張）

---

## 四、 影片合成與最終產出 (Video Rendering & Output)
負責將中介素材打包成最終的影片產品，並產生營運所需的 Metadata。

*   **核心合成工具**：**FFmpeg** (透過 Python `subprocess` 模組自動化呼叫)。

### 合成策略（兩階段）

#### 第一階段：每章節獨立產出 MP4
*   每個 `chapter_<N>.wav` 搭配對應的章節標題卡，個別產出 `<書名>_chapter_<N>.mp4`。
*   使用靜態圖片（無 zoompan），`-preset ultrafast -crf 28`，速度比 zoompan 快 **20~50 倍**。
*   **速度參考**：10 章約 30 秒完成（舊版 zoompan 需 7+ 分鐘）。
*   **章節 MP4 儲存路徑**：`Workspace/<書名>/Video/<書名>_chapter_<N>.mp4`

#### 第二階段：串流合併成完整影片
*   所有章節 MP4 透過 FFmpeg `concat demuxer` + `-c copy` 串流合併（**不重新編碼**）。
*   不論幾千章，合併步驟只需 **5~10 分鐘**，與影片總時長無關。
*   最終單一影片可達數百小時，可直接上傳 YouTube。

### 目錄結構
```
Workspace/<書名>/
  ├── RawText/       ← 原始抓取文本
  ├── CleanText/     ← TTS 用清理文本
  ├── Audio/         ← 各章節 WAV（<書名>_chapter_<N>.wav）
  ├── Video/         ← 各章節 MP4（中間暫存，<書名>_chapter_<N>.mp4）
  └── Images/        ← 各章節標題卡（<書名>_chapter_<N>.jpg，由 image_gen.py 自動產生）

Output/<書名>/       ← 最終成品
  ├── <書名>_full.mp4          ← 整本書單一影片（直接上傳 YouTube）
  └── youtube_metadata.txt     ← 章節時間戳 + AI 聲明
```

### 附屬產出
*   **`youtube_metadata.txt`**：自動生成精準章節時間戳（格式：`HH:MM:SS 章節標題`），可直接貼到 YouTube 影片說明欄，提升 SEO 與使用者體驗。

---

## 五、 全自動化任務調度與排程 (Task Scheduler)
保證所有上述模組能像工廠輸送帶一樣，一個接一個自動運作。

*   **控制中樞**：Python 主控程式 (`main_pipeline.py`)。
*   **配置驅動 (Config-Driven)**：所有變動參數（如書名、網址、聲線、TTS 引擎選擇）皆定義於 `config.yaml` 中，主程式只需讀取設定檔即可運作，不需改動程式碼。
*   **執行環境**：在 Spyder (Anaconda) 中透過 `%runfile` 執行，或由 Windows 工作排程器定時觸發。
*   **容錯與日誌 (Logging & Fallback)**：
    *   全面導入 Python `logging` 模組，所有動作與錯誤即時寫入 `Workspace/<書名>/system.log`。
    *   主控程式每步驟獨立 `try...except`，任一子模組失敗不影響後續步驟繼續執行。

## 六、 檔名命名規範（統一）

| 類型 | 命名格式 | 範例 |
|------|----------|------|
| 原始文本 | `<書名>_chapter_<N>_raw.txt` | `凡人修仙傳_chapter_1_raw.txt` |
| 清理文本 | `<書名>_chapter_<N>_clean.txt` | `凡人修仙傳_chapter_1_clean.txt` |
| 章節音訊 | `<書名>_chapter_<N>.wav` | `凡人修仙傳_chapter_1.wav` |
| TTS 暫存 part | `<書名>_chapter_<N>_tmp_part_<NNN>.wav` | `凡人修仙傳_chapter_1_tmp_part_001.wav` |
| 章節影片 | `<書名>_chapter_<N>.mp4` | `凡人修仙傳_chapter_1.mp4` |
| 章節標題卡 | `<書名>_chapter_<N>.jpg` | `凡人修仙傳_chapter_1.jpg` |
| 完整影片 | `<書名>_full.mp4` | `凡人修仙傳_full.mp4` |

> **原則**：所有中間產物使用 `_tmp_` 前綴標示，流程完成後自動清除。最終輸出檔名不含 `part`、`tmp` 等暫存標記。

## 七、 營利防禦指南 (核心心法)
要成功開啟營利，必須滿足「對 YouTube 有實質貢獻」的要求：
1.  **獨特標題與包裝：** 不要只寫「第1集」。例如：「凡人修仙傳 | 第 1-5 章：韓立的七玄門之路 | 沉浸式修仙有聲書」。
2.  **視覺敘事 (防判定重複內容)：** 每章節影片使用獨立的章節標題卡（自動產生），觀眾可清楚識別當前章節。
3.  **時間戳 SEO：** 在頻道描述區放上詳盡的章節時間戳，增加搜尋權重並提升使用者體驗。
4.  **AI 聲明：** 上傳時勾選「AI 生成內容」，並在描述欄聲明：「本內容採用 AI 輔助製作，配音與視覺皆經二次原創優化處理」。

---

## 八、 風險控制與長期策略
1.  **版權申訴風險：** 《凡人修仙傳》是熱門大 IP，作為首發測試很適合，但長期大規模搬運仍有版權風險。
    *   **策略：** 系統穩定後，建議擴展至「長尾、冷門但有死忠受眾」的仙俠小說，或是無版權爭議的作品。
2.  **增加頻道原創性 (人格化)：** 為了進一步降低「搬運感」，系統未來可擴充「AI 卷首語」功能：讓 LLM (如 ChatGPT) 根據該集內容自動生成一段 1 分鐘的「前言導讀」或「集後點評」，由 AI 語音先進行串場，這將大幅提高 YouTube 判定的原創價值。
